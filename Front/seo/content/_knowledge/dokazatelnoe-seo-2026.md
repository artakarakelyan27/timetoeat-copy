# Доказательное SEO 2026 · Knowledge Base

Источник: DrMax «Доказательное SEO 2026» (анализ утечки Google Content Warehouse API 2024 + материалов антимонопольного дела США 2020-2025).

Этот файл — не для пользователей. Это шпаргалка для всех, кто проектирует или ревьюит страницы plus-time.ru. Дополняет GIST Content Logic Skill v2.

## Главный сдвиг парадигмы

От **обратной инженерии** к **архитектурному соответствию (architectural alignment)**.

Не «обмануть алгоритм», а помочь Google построить максимально точное **цифровое досье** (CompositeDoc) для каждой страницы и для домена.

Раскрыто >2500 модулей и >14 000 атрибутов хранения данных Google.

---

## Архитектура ранжирования Google (что мы знаем точно)

Это не «один алгоритм». Это конвейер из нескольких независимых систем:

1. **Индексация и тиризация** — разбиение URL'ов по тирам качества.
2. **Mustang** — первичный скоринг и отбор кандидатов.
3. **NavBoost** — переранжирование на основе поведения пользователей. Анализирует **goodClicks** и **lastLongestClicks**.
4. **Twiddlers** — финальные корректировки результатов.

Базовые структуры данных: `CompositeDoc` (полный документ), `PerDocData`, `PageContentInfo`.

## Два столпа оценки

### Q* (Quality) — медленная, фундаментальная репутация

- `siteAuthority` — подтверждённая «авторитетность домена». Накапливается годами.
- `TrustScore` — базовый рейтинг доверия (не путать с DA от Moz).
- «Алгоритмический импульс» — когда сайт начинает расти, ему дают bonus.
- `siteFocusScore` — насколько сайт **сфокусирован** на одной теме. Размытие тематик = понижение.
- `topicAuthority` — авторитет в конкретной нише.

**Для plus-time.ru:** мы строим узкоспециализированный сайт про планирование питания — это плюс для `siteFocusScore`. Любые «расширения» в смежные ниши (фитнес, рестораны, кухонная техника) — снижают фокус.

### P* (Popularity) — быстрая, динамическая популярность

- `NavBoost` использует поведенческие сигналы из Chrome (а не только из SERP).
- `goodClicks` — клик с длительной задержкой возврата (= пользователь остался читать).
- `lastLongestClicks` — самый длинный клик в сессии (= нашёл то что искал).
- **Прямые заходы** на бренд (`Время Есть` в Yandex/Google) — сильный P*-сигнал.
- Время на сайте, количество страниц за сессию.

## Контент: ключевые алгоритмические метрики

### `contentEffort` — алгоритмическое измерение человеческого труда

LLM-модели Google оценивают «сложность воспроизведения» контента. Высокий contentEffort даёт:
- Уникальные данные, которых нет у других.
- Мультимедиа (фото, видео, схемы, интерактивы).
- Демонстрация экспертности (личный опыт, конкретные числа).

**Для нас:** компенсируем отсутствие фото:
- Интерактивные селекторы (`/chto-prigotovit/iz-togo-chto-est` — уже есть)
- Калькуляторы порций и КБЖУ
- Конкретные тайминги по дням ("понедельник 35 мин, вторник 15 мин")
- Сравнительные таблицы
- Уникальные методологические разделы (`/o-proekte`)

### `OriginalContentScore` — оценка оригинальности

Борьба с тонким и шаблонным контентом. Чёткий сигнал: если контент алгоритмически близок к существующим в индексе → пессимизация.

**Защита от этого:**
- **Anti-replicability engineering** — намеренное усложнение структуры воспроизведения.
- **Data-shift optimization** — смещение уникальности из текста в данные (таблицы, цифры).
- **Anti-template strategy** — контролируемая асимметрия страниц (не все по одному шаблону).
- Явное внедрение человеческих ограничений и ошибок (не «идеальные» статьи).

### `ContentChecksum96` — нахождение дубликатов

Google вычисляет 96-битный hash контента и находит дубликаты. Каноникализация — критична.

### `clutterScore` — замусоренность

Плохая реклама, навязчивые модалки, длинные cookie-banner'ы. Замусоренность одной страницы **размазывается на весь домен** через `isSmearedSignal`.

**Для нас:** никаких pop-up'ов «подпишитесь», никаких modals, никаких лишних interstitials. Только sticky CTA-плашка в углу — минимально-инвазивно.

## Семантика

### EntityAnnotations + Knowledge Graph

Google аннотирует страницы по сущностям из Knowledge Graph (а не только по ключевым словам).

**Для нас:** каждая страница должна явно содержать **именованные сущности**: продукты по их полному названию (не «курочка», а «куриная грудка»), кухни, диеты, техники. Ингредиенты по латинским именам в schema.org/Recipe.

### QBST (Query-Based Salient Terms)

«Запомненная» релевантность — список терминов, которые Google ожидает увидеть на странице по конкретному запросу.

**Для нас:** на странице `/spravochnik/zameny/yajco` должны быть слова из всех вариаций запроса: «заменить», «вместо», «без яиц», «аналог», «чем», «что использовать». Уже частично есть.

### Визуальная иерархия

`avgTermWeight` — размер шрифта **имеет значение**. Если ключевая фраза в `<h1>` крупнее обычного текста — она получает больший вес.

## Off-Page

### `PageRank-NearestSeeds`

Близость к «доверенным источникам» важнее количества ссылок. Один link от сайта с PageRank-Seeds-близостью 1 = тысячи ссылок от низкокачественных доменов.

**Для нас:** в первую очередь нужны упоминания на:
- food.ru, russianfood.com (формально — конкуренты)
- Сайтах диетологов, здорового питания (zdorovoe-pitanie.gov.ru)
- IT-СМИ (rb.ru, vc.ru) — для PR
- Telegram-каналах с реальным трафиком

### `queriesForWhichOfficial`

Google идентифицирует, для каких запросов сайт является «официальным» источником. Когда «Время Есть» начнёт превалировать в SERP по своему бренду — этот сигнал активируется.

### Брендовые сигналы

- Прямые заходы (типа в адресной строке).
- Брендовый поиск.
- Данные Chrome → ОЧЕНЬ силён.

## Технические сигналы

### CWV — не прямой фактор, а триггер NavBoost

`mobileCwv` и `desktopCwv` напрямую используются для «изменений в ранжировании». 

**Цели:**
- LCP ≤ 2.5s (75-percentile)
- INP ≤ 200ms
- CLS ≤ 0.1

### TTFB-per-doc

Time to First Byte, собранный **через Chrome**, важнее чем кажется. nginx + статика SSG = TTFB ~50-100ms — это хорошо.

### `isSmartphoneOptimized`

Больше, чем просто responsive. Включает:
- Touch targets ≥44px (мы соблюдаем).
- Безопасные зоны (env(safe-area-inset)).
- Скорость на мобильном (отдельный CWV-набор).

## Структура: контентные коконы

Книга описывает концепцию **матриархальной перелинковки** (mother-child structure):

- **Уровень 1: Матриарх** — pillar-страница (наша `/menyu-na-nedelyu`, `/recepty`, `/spravochnik`).
- **Уровень 2: Mixed Pages** — категории и хабы.
- **Уровень 3: Support Pages** — конкретные «атомарные» страницы (рецепт, замена ингредиента).

Авторитет течёт **снизу вверх и сверху вниз**. Все Support Pages линкуют на Матриарх. Матриарх линкует на топ-Support Pages.

**Для нас:**
- `/menyu-na-nedelyu` (матриарх) → линкует на 3 главных меню (`dlya-semi`, `pp`, `byudzhetnoe`).
- Каждое меню линкует обратно на матриарх + на 7 рецептов из своих дней.
- Каждый рецепт линкует на пары `/iz/[X]-i-[Y]` и на хаб категории.

`TopicAuthority` агрегируется по всему кокону.

## E-E-A-T в эпоху утечек

Расшифровка: **Experience** + **Expertise** + **Authoritativeness** + **Trust**.

### Experience (опыт)
Личный опыт автора. У нас нет именного автора — компенсируем:
- `/o-proekte` с прозрачной методологией ✓
- Конкретные данные: «по нашему опыту тестирования», «команда «Время Есть» проверила»
- Уникальные наблюдения от пользователей (когда появятся отзывы)

### Expertise (экспертность)  
- Источники КБЖУ из Роспотребнадзора + USDA ✓
- Точные числа, граммовки, температуры
- Объяснения «почему» (а не только «как»)

### Authoritativeness (авторитет)
- Брендовые упоминания
- Цитирования в СМИ
- Использование `Organization` schema ✓

### Trust (доверие)
- Контактная страница, email
- Прозрачные источники
- Дата обновления контента ✓ (в yaml поле `updated`)
- `dateModified` в schema.org

## Doctrine для plus-time.ru — что делать ВСЕГДА

1. **Один URL — один смысл.** Каноникал везде. Параметры сортировки → noindex.
2. **Свежесть.** На каждой странице — `lastModified` в schema + visible на странице.
3. **Сущности.** На странице о борще должны быть упомянуты: свёкла, говядина, капуста, сметана, чеснок — в составе schema.org/Recipe `recipeIngredient` И в тексте.
4. **Числа.** Конкретные цифры выгодны: 312 ккал, 25 мин, 4 порции, –348 руб. экономии.
5. **Anti-template.** В каждом архетипе — небольшая вариативность. Например рецепты с временем <20 мин показывают плашку «быстрый» в другом цвете; pp-меню имеет инфографику КБЖУ итого, бюджетное — графу «стоимость в Самокате/Лавке».
6. **Антишаблон GIST + contentEffort.** Никаких генерических вступлений. Минимум 2 уникальных смысловых узла на страницу.
7. **Перелинковка матриарх-подход.** Каждая Support линкует ввверх (на матриарха) + вбок (на 1-2 родственных) + на 2-3 связанных хаба.
8. **Никаких pop-up'ов.** clutterScore критичен. Sticky CTA — единственный наш активный элемент.
9. **Мобильный first.** Все компоненты сначала проектируются для 375px, потом для desktop.
10. **Скорость.** SSG > SSR > CSR. nginx статика. Никаких блокирующих скриптов.

## Doctrine для plus-time.ru — что НЕ делать НИКОГДА

- Авто-вставка рецептов из чужих источников без правок (ContentChecksum96 поймает).
- Дубликаты контента под разными slug'ами.
- Каскад редиректов (301 → 301 → 200).
- Большие cookie-banner'ы на пол-экрана.
- Pop-up «подпишитесь на рассылку» в первые 5 секунд.
- Маскировка контента под рекламу (или наоборот).
- Скрытый текст для роботов.
- Манипуляция кликами (CTR-боты, искусственные сессии).
- Закупка ссылок с PBN/линкоферм (PageRank-NearestSeeds такие не учтёт всё равно).

## Что использовать из второй книги (Промптоведение)

Когда будем масштабировать контент через LLM (фаза 2):

- **RAG** — LLM работает с нашими данными (из БД), не выдумывает.
- **Контентный промпт** должен включать: список сущностей, целевые QBST-термины, антишаблонные ограничения.
- **Hard Link Injection** — промпт явно требует от LLM вставить N внутренних ссылок в конкретные блоки.
- **TopicEmbeddings** — проверка релевантности генерируемого контента к кокону.
- **Промпт-валидация** — отдельный проход LLM проверяет результат на «правда/выдумка», на стоп-слова брендбука, на наличие requied entities.

См. отдельный файл `gist-prompts-pack.md` (загружен пользователем) — он совместим с этим подходом.
